# 数当てゲーム / Number Guessing Game

## 1. プロジェクト概要

Spring Bootを使用して開発した数字当てゲームのWebアプリケーションです。
ユーザーはログイン後、ゲーム画面で3桁の数字を入力し、サーバー側で正解数字と比較した判定結果を確認できます。

ログイン情報、ポイント情報、ゲーム試行回数、正解数字、ゲーム結果履歴はPostgreSQLに保存し、データベースアクセスはMyBatisを使用して処理しています。

---

## 2. 主な機能

### 会員機能

* ログイン
* ログアウト
* セッションによるログイン状態の保持
* 未ログインユーザーのゲーム画面アクセス制限
* ログイン失敗時のエラーメッセージ表示
* データベース接続失敗およびシステム例外メッセージの処理

### 数当てゲーム機能

* 3桁のランダムな正解数字の生成
* 重複しない正解数字の生成
* ユーザー入力値と正解数字の比較
* Strike / Ball / はずれ の結果判定
* ゲーム試行回数の管理
* 当日のゲーム結果履歴の取得
* 最大10回までの試行制限
* 正解時のポイント付与
* 失敗時のゲーム終了処理
* 日付変更時のゲーム情報初期化

### ポイント付与機能

正解した試行回数に応じて、ポイントを段階的に付与します。

| 試行回数     | 付与ポイント |
| -------- | -----: |
| 1回 ～ 5回  |   1000 |
| 6回 ～ 7回  |    500 |
| 8回 ～ 10回 |    200 |

### 画面機能

* ログイン画面
* 数当てゲーム画面
* ポップアップ画面
* ゲーム結果リストの表示
* 成功 / 失敗 / エラーメッセージの表示
* ログアウト確認ボタンの制御

## 6. 主な実装内容

### セッションによるログイン管理

ログイン成功時に会員情報をセッションに保存し、ゲーム画面へアクセスする際にセッション内にログイン情報が存在するか確認します。
未ログインユーザーがゲーム画面にアクセスした場合は、ログイン画面へリダイレクトします。

### 数当てゲーム処理

サーバー側で重複しない3桁の正解数字を生成します。
ユーザーが入力した数字と正解数字を比較し、以下のような結果を返します。

* 同じ数字が同じ位置にある場合は Strike
* 同じ数字が異なる位置にある場合は Ball
* 一致する数字がない場合は `はずれ`

### ゲーム進行状態管理

ユーザーのゲーム回数とゲーム終了状態をDBに保存します。
正解した場合、または最大試行回数である10回に達した場合は、ゲーム終了状態に変更します。

### 日付基準のゲーム初期化

最終更新日と現在日付を比較し、日付が変更された場合は以下の値を初期化します。

* ゲーム終了フラグ
* ゲーム試行回数
* 正解数字

### ゲーム結果履歴表示

当日のゲーム結果を取得し、ゲーム画面にリスト形式で表示します。
ユーザーは過去の入力値と判定結果を確認できます。

### ポップアップ画面処理

ゲーム終了後、ポップアップ画面で成功または失敗メッセージと獲得ポイントを確認できるように実装しました。

### 例外処理

ログイン処理およびゲーム処理中に例外が発生した場合、ユーザーにメッセージを表示するように実装しました。
ゲーム処理後にリダイレクトが必要な場合は、`RedirectAttributes`を使用して画面データを渡します。

## 8. トラブルシューティング

### Redirect後にデータが保持されない問題

`redirect`を使用した場合、`Model`に保存したデータが次のリクエストまで保持されない問題がありました。
この問題を解決するために、`RedirectAttributes.addFlashAttribute()`を使用して一時的なデータを渡すように修正しました。

### ゲーム結果リストのNullPointerException

ゲーム結果リストが存在しない状態で最後の結果を取得しようとしたため、`NullPointerException`が発生しました。
リストが`null`または空でないかを確認してから、最後のデータを取得するように修正しました。

### JavaScriptのポップアップ関数が認識されない問題

ポップアップ呼び出し時に関数が定義されておらず、`popup is not defined`エラーが発生しました。
JavaScript関数の定義位置とDOM読み込み後の実行順序を整理して解決しました。

### ログアウト確認ボタンがクリックできない問題

ログアウト確認エリアが他の要素と重なり、ボタンが正常にクリックできない問題がありました。
CSSの配置と`z-index`を調整し、ボタンが正常にクリックできるように修正しました。

## 9. 学んだこと

### Spring MVCのリクエスト処理の流れの理解

Controllerでリクエストを受け取り、Serviceでビジネスロジックを処理した後、DAOを通じてDBにアクセスする流れを実装することで、Spring MVCベースのWebアプリケーション全体の構造を理解することができました。
特にログイン処理、ゲーム画面への遷移、ゲーム実行処理のように、機能ごとにリクエストURLを分けることで、Controllerの役割と責任を整理する方法を学びました。

### Controller、Service、DAOの役割分担

最初は例外処理やゲーム判定ロジックをどの階層に置くべきか迷いましたが、実装を進める中で各階層の役割を区別できるようになりました。

* Controller: リクエスト処理、画面遷移、セッション確認、メッセージの受け渡し
* Service: ゲーム初期化、数字判定、ポイント計算、ゲーム終了判定
* DAO: MyBatisを利用したDB検索および更新

これにより、単に機能を動作させるだけではなく、保守しやすい構造でコードを分けることの重要性を学びました。

### セッションによるログイン状態管理

ログイン成功時に会員情報をセッションに保存し、ゲーム画面へアクセスする際にセッション内にログイン情報が存在するか確認するように実装しました。
これにより、HTTPリクエストは基本的に状態を保持しないため、ログイン状態のようなユーザー情報を維持するにはセッションを活用する必要があることを理解しました。

また、未ログインユーザーが直接ゲーム画面のURLにアクセスした場合、ログイン画面へ遷移するように処理することで、画面アクセス制御の基本的な流れを経験しました。

### RedirectとForwardの違いの理解

ゲーム実行後に画面を再表示する過程で、`redirect`を使用すると`Model`に入れたデータが次のリクエストまで保持されない問題を経験しました。
これを解決するために、`RedirectAttributes.addFlashAttribute()`を使用して、リダイレクト後にも一時的なデータを渡せるように実装しました。

この過程を通じて、`return "viewName"`方式と`redirect:/url`方式の違いを理解し、リダイレクトが新しいリクエストを発生させるという点を、実際の問題解決を通して学びました。

### DTOを利用した画面データの受け渡し

ゲーム情報、ポイント、試行回数、結果リスト、メッセージなどを`GameDTO`と`ResultDTO`に格納し、Controller、Service、JSPの間で受け渡しました。
これにより、複数のデータを個別に渡すよりも、DTOにまとめて渡す方法のほうが画面処理や保守に適していることを学びました。

また、DTO内部にリスト形式の結果履歴を含めることで、オブジェクトの中に他のオブジェクトの一覧を持たせて画面へ渡す構造も経験しました。

### MyBatisを利用したSQLの分離

Javaコード内にSQLを直接記述せず、MyBatisのMapper XMLファイルにSQLを分離して記述しました。
これにより、SQLとJavaのビジネスロジックを分離する方法のメリットを理解しました。

特に会員認証、ポイント情報取得、ゲーム結果保存、当日の結果履歴取得といった機能をMapper単位で分けることで、DBアクセス処理を管理する方法を学びました。

### PostgreSQLとJavaアプリケーションの連携

Spring BootアプリケーションからPostgreSQLデータベースへ接続し、MyBatisを通じてデータを検索・更新する流れを実装しました。
これにより、Webアプリケーションは単に画面を表示するだけではなく、DB上のユーザー状態やゲーム進行情報を基準に動作するという点を理解できました。

### ゲーム状態管理ロジックの実装

数字当てゲームは単純な入力比較だけではなく、ユーザーごとに現在のポイント、ゲーム回数、正解数字、ゲーム終了状態を継続的に管理する必要がありました。
これを実装する中で、状態をDBに保存し、リクエストが来るたびに現在の状態を再取得して処理する方式の重要性を学びました。

また、正解した場合と10回失敗した場合を区別してゲーム終了フラグを変更し、日付が変わった場合はゲーム情報を初期化するロジックを実装することで、状態に基づいたロジック処理の経験を積みました。

### 数字判定アルゴリズムの実装

ユーザーが入力した3桁の数字とサーバー側で生成した正解数字を比較し、Strike、Ball、はずれ の結果を判定するロジックを実装しました。
これにより、単純な条件分岐だけではなく、配列と繰り返し処理を活用して、位置と数字の一致判定を行うアルゴリズムを自分で作成することができました。

また、正解数字を生成する際に重複した数字が出ないように処理することで、ランダム値を生成した後に検証する方法も経験しました。

### 例外処理の流れの理解

ログイン処理やゲーム処理中に、DB接続失敗や予期しないエラーが発生する可能性を考慮して例外処理を実装しました。
Controllerで例外発生時にユーザーへメッセージを表示するように処理することで、単にコンソールにエラーを出力するだけではなく、ユーザーが理解できるメッセージを伝えることの重要性を学びました。

また、`try-catch`をどこに記述すべきかを考える中で、例外処理も階層ごとの役割に合わせて設計する必要があることを理解しました。

### NullPointerExceptionの原因分析と防御的なコード作成

ゲーム結果リストが存在しない状態で最後の結果を取得しようとして、`NullPointerException`が発生する状況を経験しました。
これにより、オブジェクトやリストを使用する前には、`null`でないか、空のリストでないかを確認する必要があることを学びました。

特に結果履歴が存在しない場合と存在する場合の流れを分けることで、例外が発生しないように防御的にコードを書く習慣の重要性を感じました。

### JSPとJavaScriptを利用した画面制御

JSPでサーバーから渡されたデータを画面に表示し、JavaScriptとjQueryを利用して、ボタンクリック、メッセージ表示、ポップアップ表示、ログアウト確認機能を実装しました。
これにより、サーバーサイドから渡されたデータとクライアントサイドのイベント処理が連動して動作する流れを理解しました。

また、JavaScript関数の定義位置やDOM読み込みのタイミングによって、`popup is not defined`のようなエラーが発生する可能性があることを経験し、スクリプトの実行順序の重要性を学びました。

### CSSレイアウトとイベント衝突の解決

ログアウト確認ボタンが他の要素と重なってクリックできない問題を解決することで、画面要素の配置が実際の機能動作にも影響することを学びました。
単に画面を見やすく作るだけではなく、ユーザーが正常にクリックし操作できる構造でCSSを作成する必要があることを理解しました。
